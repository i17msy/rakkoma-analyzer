# ラッコM&A アナライザー

## プロジェクト概要

ラッコM&A（rakkoma.com）はオンライン事業の売買プラットフォーム。YouTubeチャンネル等が
匿名で売買され、Google Adsense月次収益・登録者数・再生数などが確認できる。

**目的**: 掲載案件を網羅的に収集・構造化し、**LLMが「再現性」を主軸に一次評価** →
人間はダッシュボードで横断レビューするだけ、という状態を作る。単なる新着スクレイパーでは
なく「本家より使えるアナライザー」を目指す。

### なぜ再現性重視か
MLBパイプライン（自前YouTube運営）の参考になる「継続収益があり自分で再現できるチャンネル」を
発見・蓄積したい。属人性が低く・自動化/外注しやすく・ロングテールに稼ぐモデルを高く評価する。

### 北極星（最終ゴール / ワンストップ）
```
発見(ラッコ) → 一次評価(再現性 × 能力適合 × リスクフラグ) → YouTube候補照合(確率的)
            → ベンチマーク特徴のテンプレ化 → 動画パイプラインの横展開指標
```
ラッコの匿名データに「YouTubeプラットフォームの実ファクト」を突き合わせて実チャンネルを**絞り込み**、
特定できた一部について **売却の実否・運営者交代・停止復活** を検証する。これはMLBで人力でやった作業の
*部分的な*自動化で、次のベンチマークchを見つけた時に「ベンチマークとすべき特徴」をテンプレ化し、稼働中の
動画パイプラインを新ニッチへ横展開する際の指標にする。このデリバリーまでをワンストップで提供するのが
RakkoAnalyzer の現時点の目標。

> ⚠️ 技術メモ（将来実装・**確度を正直に**）: これは**決定論的な"特定"ではなく確率的な候補照合**。
> ドキュメントを確信的に書くと実装を「確実に特定できる前提」で設計してしまうため、ここは calibrated に記す。
> - ラッコの 登録者数・投稿数・運営開始時期・ジャンル は**弱い指紋**。YouTube Data API で
>   登録者数±・動画本数±・開設時期・ジャンルから候補を絞れるが、限界が大きい:
>   (1) 匿名化で**最強の識別子(ch名/URL)が落ちている** (2) ニッチ内に近い登録者数・本数の類似chが多数
>   (3) **月次収益系列はYT APIで直接照合できない**（過去の月次再生/収益を返さない。収益×RPMで再生数を
>   逆算しても突合は間接的・弱い） (4) 売却後に改名/非公開/削除される個体もある。
> - よって **多くの案件は特定不能 or 低確度に留まる**前提で設計する。出力は「**信頼度付き候補リスト**」とし、
>   **「特定できない/曖昧」を一級の結果**として扱い、**最終確認は人間(human-in-the-loop)**。下流ロジックは
>   確実な特定を前提にしない。
> - 特定できた一部のみ、出品前後のアップロード/ブランド変化から売却の実否を推定。「2度以上売却」等の深い
>   検証はその先（MLBベンチは*目視で*最低2回売却済みと判明＝人力なら可能、自動化はあくまで候補提示の補助）。
>
> **用途は「厳密な穴埋め」→「近似ベンチマーク化」へずらす**: 本来の狙いはラッコが欠落させる情報
> （再生数トレンド・過去の運営/売却履歴 等）の補完で、人力では「このchは2度売却済み」まで掘れた。だが
> 上記の確度の低さ＋系列の直接照合不可で *多くは厳密に埋まらない*。よって per-listing の厳密な復元では
> なく、「**再現すべき勝ちパターンの近似リファレンス**」を作る方向へ用途を寄せる。到達可能な天井が用途を
> 再定義する形で、下流の「ベンチマーク特徴のテンプレ化」と**確度レンジが一致**するので、この用途変更は
> むしろ整合的。実装も「厳密な穴埋め前提」で組まない。

## アーキテクチャ（3層）

```
収集・列挙 ─────────────────────────────────────────────
  scrape_rakkoma.py   RSSフィード駆動の新着監視デーモン（30分間隔・Slack通知）
  enumerate_listings.py  販売中一覧巡回 + サイトマップで直近Nヶ月のクローズを列挙
  backfill.py         enum対象を巡回取得 → 詳細パース → SQLite格納（礼儀遅延つき）
        │
格納 ───┼───────────────────────────────────────────────
  storage.py (SQLite: data/rakkoma.db)
     listings      … 最新スナップショット（メトリクス列込）
     price_history … 価格・ステータス変化の履歴
     evaluations   … LLM評価（listing_idごと最新）
        │
分析・レビュー ──────────────────────────────────────────
  metrics.py     定量メトリクス算出（回収月数・収益安定度・登録者あたり利益）
  analyze.py     再現性重視のLLM評価（Claude / 構造化ツール出力）
  dashboard.py   静的HTMLダッシュボード生成（data/dashboard.html）
```

## サイト仕様（確認済み）

- **RSSフィード**: `rakkoma.com/project/list/feed`（Atom・直近約50件のスライディングウィンドウ）
- **サイトマップ**: `sitemap_listings.xml` に全案件URL（lastmod付き・全件列挙が可能）
- **詳細ページ**: ログイン不要。`<th/td>` テーブルから各フィールドを正規表現で抽出
- **ステータス判別**（詳細ページHTML内バナー）:
  | 状態 | マーカー | 付随情報 |
  |---|---|---|
  | 募集中 | バナー無し | — |
  | 成約済み | `この案件は成約済みです` | **`成約期間：N日`**（売れるまでの日数＝需要シグナル） |
  | 受付終了 | `この案件は交渉の受付を終了しています` | — |
- **成約価格は非開示**（検証済み 2026-06）。成約済みでも表示価格は「希望売却価格」のまま。
  取れるのは **希望価格 + 成約期間 + 成約/受付終了の別** まで。実取引額は不明 → 市場分析は
  「価格回帰」ではなく「希望価格を所与とした需要・成約速度モデル」になる（後述）。
- **robots.txt**: `Crawl-delay: 10` を要請 / `Scrapy` は全面禁止。UAは通常ブラウザ風にし礼儀遅延を入れる

### 抽出フィールド
案件名 / カテゴリ / 希望売却価格 / 評価倍率(=回収月数表記) / 売上・利益（直近/平均/最高）/
運営開始時期 / 運営状況 / 投稿数 / フォロワー・登録者数 / 収益モデル / コンテンツの性質 / 説明文 /
**公開日(掲載日)** / **更新日** / ステータス / 成約期間 / **月次収益系列(12ヶ月)**

### ⭐ 月次収益系列が真実の入力（重要）
詳細ページのグラフ部品 `<list-content-area-chart :data-list="[...]">` に**直近12ヶ月の月別収益**が
丸ごと埋め込まれている。3点要約（直近/平均/最高）は**壊れる事がある**（例: ID19490は直近=平均=最高が
全て同値の degenerate 値で、LLMが「単月のみ」と誤診して虚偽の弱みを生成していた）。よって
**系列を一次入力**にし、3点要約は系列が無い時のフォールバックに降格。全メトリクスを系列から再算出。

### 日付の整理（混同注意）
- **公開日(listed_at)**: ラッコに出品された日。**滞留日数 = 今日 − 公開日**で「在庫滞留」を算出（募集中にも効く需要シグナル）
- **更新日(updated_at)**: 出品情報の最終編集日（値下げ等の痕跡）
- **運営開始時期(start_date)**: 事業/チャンネルが始まった時期（粗い・年月）
- **成約期間(deal_days)**: 売れるまでの日数（成約済みのみ）

### 定量メトリクス（metrics.py / 系列ベース）
- 回収月数（直近/平均利益ベース）… price ÷ 利益
- 平均利益 = **収益化期間の平均**（立上げ前ゼロ月を除外）/ 直近・最高・最低
- 勢い（旧:安定度）= 直近 ÷ 平均（1超=上振れ / 1未満=下振れ）。一覧では↗→↘の矢印で表示
- **変動 CV** = 標準偏差 ÷ 平均（収益化期間）… 真のブレ。0近い=安定 / 0.5超=スパイキー(バズ依存)
- **トレンド%** = 収益化期間 直近3 vs 最初3 … 右肩上がり/下がり
- 月数 / 収益化月数 / 立上げ期間（運営 − 収益化）
- 滞留日数 = 今日 − 公開日 / 登録者1000人あたり月利益

### ⭐ リスク/実績フラグ（決定論・metrics.py + dashboard）
系列の"形"を一言のフラグに圧縮（ラッコ本体より高度に・シンプルに）。優先順にアイコン表示:
| フラグ | 条件 | アイコン |
|---|---|---|
| 収益不明（判断不能） | 月次系列なし＝持続性/トレンドを検証不能 → リスク×0.7 | ❓ |
| 収益ゼロ | 系列が全ゼロ＝収益実績ゼロ → リスク×0.7 | 🚫 |
| 運営乖離（再販/移管疑い） | 収益化月数 − 運営月数 ≥ 3 | ⚠️ |
| 停止復活歴 | 系列の途中ゼロ | ⛔ |
| 急成長×ピーク売り抜け | トレンド≥+50% かつ 直近/最高≥0.8 | 🚀 |
| ピーク売り | 直近/最高 ≥ 0.9 | 🔝 |
| 立上げ初期 | 収益化月数 ≤ 6 | 🌱 |
| 高変動 | CV ≥ 0.5 | ⚡ |
| 下降トレンド | トレンド ≤ −25% | 📉 |
| 実績安定 | 収益化≥12ヶ月 かつ CV<0.3 かつ 非下降 | ✅ |

### ⭐ 履歴インサイト（4指標を1判定に合成 / dashboard）
運営・収益化・立上げ・乖離 を1つのライフサイクル判定に集約（人が頭で合成しない）:
移管疑い(乖離≥3) → 🌱新規(運営≤6ヶ月) → 📈成長期(≤1.5年×収益化未飽和) → 🏛成熟(2年超×継続収益) → ✓確立

## 評価仕様（analyze.py）— 脱ジャンル先験・能力適合の2軸

**探索方針**: スポーツ解説等の既知パターンを高評価する**ジャンル先験を抜き**、ジャンル非依存の
構造的再現性だけで評価する。狙いは「自分が思いつかなかったジャンルの再現性」の発見（探索）。
既知のスポーツ解説の活用（exploitation）は並走させ、置き換えない。探索は偽陽性が増えるため、
高スコア×想定外ジャンルの少数に人間の精査を集中投下する。

LLM（既定 `claude-sonnet-4-6`）が構造化ツール出力で以下を提出:
- **構造的再現性スコア(各1-5・ジャンル非依存)**: 再現性(属人性の低さ・自動化/外注可能性) /
  収益持続性 / 割安度 / 成長余地
- **能力適合 `capability_fit`(1-5)**: 自分の「**顔なし・AIナレーション動画（+隣接フォーマット）**」
  パイプラインで再現可能か。物販やWebサービス等、構造的に自動化可能でも自分の生産能力では
  作れないものはここで沈む（**ジャンルではなく能力で切る**）。
- **総合スコア** = 構造スコアの重み付け和 × (capability_fit / 5)。能力適合は加点ではなく**乗数ゲート**。
- ジャンル分類 / 強み / 弱み・リスク / 再現メモ / 総合判定（買い・様子見・見送り）/ 一言サマリ

**検証基準(falsifiable)**: 全件評価の上位に「**想定外ジャンル かつ 能力適合が高い**」案件が複数
入れば脱バイアスは機能している。物販等で埋まるなら能力適合ゲートの不全、スポーツ解説で埋まる
なら (a)本当に構造的に優れる か (b)評価に隠れ先験が残る のサイン。

## 新着監視デーモン（scrape_rakkoma.py）— SQLite一元化・評価ベース厳選通知

RSS駆動（30分間隔）。新着を **SQLiteに一元化**し、動画系はその場でLLM評価して**厳選通知**:
- 全新着 → 詳細取得 → metrics.compute → **storage.upsert_listing**（系列・フラグ込みでDBへ）
- 動画系（YouTube判定）→ **analyze.evaluate**（LLM）→ evaluations 保存
- **厳選通知（config）**: 買い/様子見 × 適合 ≥ `NOTIFY_MIN_FIT`(4) × 総合 ≥ `NOTIFY_MIN_OVERALL`(2.5)
  のみSlack（「🦦 要確認の新着 — ダッシュボードで ID◯◯ を精査を」）。見送り・非動画・物販は通知しない
- 新着があれば **dashboard.main() で再生成**（ライブ更新）
- 旧 Tier1/Tier2 キーワード通知・data/listings JSON 保存は廃止（評価ベース＋SQLiteに一本化）
- 専用 **Dockerfile**（python:3.12-slim ＋ requests/anthropic のみ）＋ run_container_rakkoma.sh で常駐
  （rakkoma:latest を自動ビルド・--restart unless-stopped・単体運用＝/root/work 不要）
- ※要 ANTHROPIC_API_KEY（評価）/ SLACK_WEBHOOK_URL_RAKKOMA（通知）。鍵が無ければDB保存のみで継続
- **重複ポーリング防止（アイドル）**: 同一リポジトリを複数コンテナがマウントして共存すると
  両方が seen を奪い合い・二重通知になる。daemon は起動時に自分のホスト名(=コンテナ短縮ID)を
  `data/idle_hosts.txt`（または環境変数 `RAKKOMA_IDLE=1`）と照合し、該当すれば**監視せず待機**。
  → 残したい既存コンテナを restart してもコードが自分でアイドル化（凍結が永続）。監視は1コンテナに収束

### ⭐ DBは単一実体（複数コンテナ共存時の注意）
- 各コンテナは `/home/masaya/rakkoma` を**バインドマウント**するので、`data/rakkoma.db` は
  **ホスト上の物理1ファイルを全員が共有**する。コンテナごとにDBが分裂することはない（"本物"は常にこの1個）。
- journal_mode=`delete`＝**書き手は1人ずつ**（ファイルロック）。アイドルガードで監視を1コンテナに収束させる
  ことは「二重通知の防止」と同時に「**単一SQLiteを2プロセスが奪い合う(`database is locked`)のを防ぐ**」意味も持つ。
- 読み取り（分析クエリ/ダッシュボード生成）は書き込みと並行しても安全。`data/dashboard.html` は
  DBから毎回作り直す**派生物**なので「最後に書いた方」になるだけで整合性に影響しない。
- 含意: アイドル側コンテナから流す操作は**読み取りに留める**。書き込みバックフィルを流すなら稼働側の
  ポーリングと重ならない瞬間に（busy_timeout 任せでも可だが短時間トランザクションで）。

### ⭐ クラウド連携: R2 死活監視＋DBバックアップ（r2.py）
Cloudflare R2（S3互換・**エグレス無料**）を専用バケット `rakkoma-analyzer` に使う。observer が毎ポーリングで `_pulse()`:
- **ハートビート**: `heartbeat/observer.json` を上書き（`ts`/`host`/`interval_sec`/`stale_after_sec`/`listings`/`evaluated`）。
  クラウド側（別Claude等）が読み、`now - ts > stale_after_sec`（=interval×3≒90分）なら**死亡判定**。新着0でも必ず打つ。
- **DBバックアップ**: 1日1回、稼働中でも整合する SQLite online backup でローカル `data/backups/rakkoma-YYYYMMDD.db` を作り、
  R2 `backups/` へ未アップなら上げる。ローカル/R2 とも直近 `BACKUP_RETENTION`(14)日分のみ保持。
- 認証は env。**account/key/secret はMLBと共通の汎用 `R2_ACCOUNT_ID`/`R2_ACCESS_KEY_ID`/`R2_SECRET_ACCESS_KEY`**、
  **バケットだけ `R2_RAKKOMA_BUCKET`（ホスト）→ コンテナ内 `R2_BUCKET` に変換**して分離。**未設定なら全て no-op**
  （ただしローカル日次バックアップだけは動く）。依存は boto3（Dockerfileに追加・要再ビルド）。
- 保護階層: **論理事故**=ローカル/R2両方が守る。**WSL/ディスク障害**=R2（オフディスク）が本命。
  さらに堅くするならR2を**月次でクラウド間コピー**や、ホストから `/mnt/c` への週次コピーも可。
- heartbeat は **MLBの cloud routine 互換**（`last_poll`(epoch)/`last_poll_jst`/`host` を同梱）。
  独自の monitor を作らず、MLBと同じ3-state routine を**rakkoma用にクローン**して使う（下表）。

#### cloud routine による死活監視（rakkoma・MLBパターンに準拠・✅稼働中）
| 項目 | 値 |
|---|---|
| routine ID | `trig_01Bt5MzKGD1NRw5BbT4AzWyQ`（2026-06-29 作成・有効） |
| 監視対象 | R2 `rakkoma-analyzer` / `heartbeat/observer.json` |
| スケジュール | `45 5,23 * * *` UTC = JST 08:45 / 14:45（MLBと同じ・毎日） |
| **しきい値** | **5400秒（90分）= ポーリング30分 × 3欠落**（MLBの1800とは別値） |
| 通知先 | rakkoma の Slack チャンネル（厳選通知と同じ webhook） |
| 監視ロジック | 3-state（正常:silent / 停止疑い:⚠️ / 監視系異常:🔍） |

- State 2（`now - last_poll > 5400` かつ fetch成功）→ ⚠️ `[要確認] rakkoma observer が停止している可能性`
- State 3（R2 fetch失敗）→ 🔍 `[監視系異常] R2読み取りに失敗（observerとは無関係の可能性）`
- ※ heartbeat に `stale_after_sec`(5400) を同梱しているので、routine 側でハードコードせずこの値を読んでもよい

## 市場シグナル分析（本家を超える核心）

LLM評価は「自分の物差し（再現性価値）」の関数。成約データは「市場の客観的値付け」の関数。
**両者の乖離こそ最も情報量が高い** ——「LLM高評価なのに市場では売れ残った/安く速く売れた」案件は、
市場の見落とし（割安）か、LLM評価の見落とし（隠れリスク）かのどちらかで、両方とも示唆的。

実取引額は非開示なので「価格回帰」はできない。代わりに **希望価格を所与とした需要モデル** を組む:
- **成約 vs 受付終了**（売れたか）… 二値の需要シグナル
- **成約期間（N日）**… 需要の強さ・価格妥当性の連続シグナル
- 説明変数: 回収月数・収益安定度・ジャンル・登録者規模・収益モデル
→ 「この希望価格・このメトリクスなら市場は何日で吸収するか」を推定し、LLM評価との乖離を検出する。
これは蓄積データを持つ我々にしかできず、本家の一覧では不可能な分析。

> 接続: ベンチマークとなるMLB翻訳解説系の類似案件が過去いくらで・何日で売れたかは、自前MLB
> チャンネルの「市場基準の内在価値」を逆算する材料にもなる（発見装置 兼 値付け装置）。

### 乖離分析の結論（クローズ362件をLLM評価 / 2026-06-29）
LLM評価と市場の実挙動（成約/取り下げ）を突合。**核心の発見**:
- **LLM総合 ≠ 市場成約率（山型・単調でない）**: 総合≥3で成約27% / 2-3で45% / <1で24%。
  「我々の高評価ほど売れる」にはならない
- **山型の正体はほぼ価格**: 高総合の非成約は希望¥170万 vs 成約¥125万（回収は同じ~6-7ヶ月）。
  = LLM評価は正しく良案件を拾うが、**売り手のオーバープライスで売れ残る**。品質でなく値付けの問題
- **総合は「適合ゲートした個人的物差し」なので市場（多様な買い手）とズレて当然＝正常**。市場の支配項は
  バリュー（価格/回収月数）。再現性は弱く効く（再現性4→成約39% vs 再現性2→18%＝非属人ほど転売しやすい）
- **🔑 乖離＝機会 ＝ 🎯交渉ターゲット**: 「我々の高評価 × 市場が高値で拒否（売れ残り）」＝良いのに強気価格で
  売れていない案件。活性での定義(dashboard): **募集中 × (買い/様子見) × 総合≥2.0 × 持続≥3 × 滞留≥30日**（13件）
  - **精緻化の経緯**: 当初「総合≥2.0×滞留」だけだと、見送りや**収益崩壊中(持続≤2)の案件**まで拾って混乱した。
    ID21998（2026アニメ化カタリスト賭けだが直近月収3,354円・−55%で収益崩壊中）を開けて判明 — レッドゾーン
    総合×様子見×🎯の正体は「価格ネック」でなく「中身が崩れている」案件だった。**交渉＝価格で直る**を担保する
    ため **見送り除外＋持続≥3（収益が健全）** を条件に追加。持続≤2は価格では直らないので🎯から外す（様子見
    としては残し情報は失わない）。フラグ同士は独立だとノイズが出る＝「動いていない×追う価値がある」の積が要る
- **セラー側裏付け**: 良いchでも強気価格だと売れ残る → 自分の売却も「12ヶ月安定実績 × 割安寄り価格」が最速

## 将来構想: セラー側への反転（build-to-sell / 自分の出口戦略）

買い手評価の機械はそのまま売り手ツールに反転する。自分が収益チャンネルを作れたら、本アナライザーで
**売り時判定・販売価格・販売戦略**を組める。追加は主に「セラービュー（同じデータの再フレーム）」＋需要モデル。

- **販売価格**: 需要モデル＝値付けエンジン。市場コンプから「この登録者/収益/メトリクスなら希望◯円で
  P(売れる)・成約◯日」を逆算。EDA実測の"売れる価格帯"（回収~12ヶ月・¥40万級）が基準
- **売り時判定（非自明）**: フラグ/軌跡が反転するが、データは「素朴な天井売り」を否定する。
  🚀急成長×ピークの成約率26%＝最低（買い手は天井を警戒）、✅実績安定42%＝最高。
  → 最適は「ランプ天井で売り抜け」ではなく「**12ヶ月の安定実績を作ってから売る**」。
  アナライザーにしか出せない売り時インサイト
- **販売戦略**: LLMの 強み＝出品アピール / 弱み・リスク＝買い手が突く所（先回りで潰す）/
  再現メモ＝買い手デューデリの想定問答
- **対称性**: 高スコアが付くch（非属人・AI/テンプレ化・安定収益）＝そのまま"売れる"設計。
  アナライザーは **build-to-sell の設計図**にもなる
- **調整**: 「適合（自分が作れるか）」は売り手側では無関係＝この軸だけ役割が外れる。売り手版は
  「再現性（買い手への訴求）× 需要モデル × フラグを"売りシグナル"として読む」
- **環の回収**: 発見/買う → 市場を評価 → 自分の資産を値付け・売り時判定。MLB/今後のchに内蔵の
  出口戦略ツールが付く（"founder legacy as timed asset" を時間軸で値付けする道具）

## 設計メモ・既知の注意点

- **データ非定常性**: サイトマップ15,592件のうち約9,300は2024-11-06の一括移行痕跡。移行前後で
  記録方式・カテゴリ粒度が非互換の恐れ。直近6ヶ月（2,240件）に絞る判断はこの汚染回避でも正しい。
  需要モデルのサンプル拡大で移行前データに手を伸ばす場合は、**フォーマット互換性の検証を一段挟む**。
- **デーモンの静かな失敗**: scrape_rakkoma.py は MLB の watch 系と同型のポーリングデーモン。
  同じ脆弱性を持つ——RSS書式変更で取りこぼし／プロセス死で新着逃し／観測失敗起因の誤通知。
  MLB側で作る死活監視（心拍・3状態分離）を**共通化して両方に効かせる**のが筋。ただし2例での
  共通基盤抽出は過剰一般化リスクがあるため、当面は緩い再利用に留め、3例目が来たら抽象化する。
- **隣接フォーマットの調査は本プロジェクト対象外（備忘）**: 能力適合の定義域に「隣接フォーマット
  （顔なしShort／ナレーション系ポッドキャスト等、同一パイプラインから届くもの）」を含めるが、
  *どんな隣接フォーマットが現に有望か* の情報収集は別課題。おそらく別種のスクレイピング/調査手法の
  確立が要り、ラッコ分析では直接は扱わない。当面 capability_fit の判定は「顔なしAIナレーション動画」
  を主軸に運用し、隣接フォーマットの知見が貯まり次第 LLM プロンプトに反映する。
